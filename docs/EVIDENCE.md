# Evidence — FileGuard

## Overview

This document catalogs the screenshots and artifacts that demonstrate
FileGuard's functionality. All screenshots are real outputs from the tool
running on local sample files.

## Screenshots

### 1. Baseline Creation

![Baseline Creation](../screenshots/baseline-creation.png)

**What it shows**: FileGuard scanning `monitored_files/` and computing
SHA-256 hashes for each file. The output displays the number of files
scanned, the first 16 characters of each hash, and confirmation that the
baseline was saved to `baseline.json`.

**Cybersecurity relevance**: Establishing a trusted baseline is the
foundation of all file integrity monitoring. Without a known-good reference,
tamper detection is impossible.

### 2. Integrity Check

![Integrity Check](../screenshots/integrity-check.png)

**What it shows**: FileGuard comparing current file hashes against the
saved baseline. The summary shows how many files are unchanged, modified,
new, or deleted, followed by a detailed report with severity classification.

**Cybersecurity relevance**: This is the core detection step. FileGuard
identifies exactly which files changed and classifies the type of change,
enabling a targeted response.

### 3. Generated Text Report

![Generated Report](../screenshots/generated-report.png)

**What it shows**: The `integrity_report.txt` file generated by FileGuard.
It contains a structured summary, details of changed files, and recommended
defensive actions.

**Cybersecurity relevance**: Documentation is critical in incident response.
A clear, timestamped report provides evidence that can be reviewed,
investigated, and preserved for later analysis.

### 4. JSON Report

![JSON Report](../screenshots/json-report.png)

**What it shows**: The `integrity_report.json` file with structured data
including scan summary, file details, severity, and recommendations.

**Cybersecurity relevance**: Structured output enables integration with
automation tools, dashboards, and SIEM systems. JSON formats allow security
data to be consumed programmatically.

## Baseline Artifact

The file `baseline.json` contains the SHA-256 hashes of all monitored files
at the time of baseline creation. Example:

```json
{
  "app_settings.txt": "bca47bee9ee8b0811cf88815117955134a04bbf160f2cac49087aace385c27df",
  "sample_config.txt": "d20cf268b5b08bb5f60b5b63f4faf6d15a393a9d7c10f91052177281d3b2abab",
  "user_data.txt": "2d40de198ca2be26f9e1fdf8343b3c8e52071d72a6172f717b18e319028677cf"
}
```

## Sample Files

Three sample files in `monitored_files/` were used for testing:

- `sample_config.txt` — example configuration
- `user_data.txt` — example user data
- `app_settings.txt` — example application settings

These are harmless text files containing sample data. They simulate the type
of files that might require integrity monitoring in a real environment.
